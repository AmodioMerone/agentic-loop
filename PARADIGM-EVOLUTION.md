# Paradigm Evolution Log

A running log of how the multi-agent coordination paradigm in this repo
evolved, what each iteration broke, and what the next iteration fixes.

## Iteration 1 — Naive shared markdown (deprecated)

**Design**: every agent appends free-form markdown to three shared files
(`shared-log.md`, `file-changes.md`, `issues.md`) and to per-agent inbox
files (`inbox/<agent>.md`). Messages and state were intermixed.

**Inspired by**: blackboard architecture (validated — +13-57% task success
over RAG in 2026 benchmarks).

**What broke under stress**:
- **Multi-writer races**. Two agents in parallel doing `Edit append` on
  the same file: each does Read → mutate → Write based on its own snapshot.
  Last writer wins → lost messages.
- **Mutable issues**. Flipping `[OPEN]` → `[RESOLVED]` inside a shared
  file is a read-modify-write that races with any other update.
- **Broadcast reads**. Every agent re-read the last N lines of the log
  every turn → O(N × agents × turns) wasted context.
- **Not queryable**. Filtering "open backend issues" required parsing
  free-form markdown each time.
- **No event identity**. Couldn't reference "the change agent X made at
  10:15" without quoting the whole line.

## Iteration 2 — Event sourcing + per-agent journals + state files

**Design**:

1. **`journal/<agent>.jsonl`** — append-only JSON-Lines, **single-writer**:
   only `<agent>` writes to its own journal. No two agents ever touch the
   same file → zero race condition regardless of parallelism.
2. **`state/issues/<ISS-id>.json`** — one file per issue, mutable, with
   a single-owner convention (the current `assigned-to`). Carries an A2A-style
   task state machine.
3. **`view/`** — derived markdown views (`log.md`, `issues.md`,
   `file-changes.md`, `inbox-<agent>.md`) regenerated by a tool from the
   journals + state files. **Agents read views, write journals.**
4. **Messages as events**, not files. A message from A to B is an event
   `{kind: "message", from: "A", to: "B", ...}` in A's journal. B's
   "inbox" is the query: all events with `to == B`.

**Borrowed from**:
- **A2A** (Google): task state machine (`submitted` / `working` /
  `input-required` / `blocked` / `completed` / `failed` / `canceled`),
  Agent Cards for discovery.
- **LangGraph**: typed structured state.
- **Event sourcing** (CQRS): immutable log + derived projections.
- **Claude Code Agent Teams**: peer-to-peer + shared task tracking
  (we sidestep the locking they need because of single-writer journals).

**Properties gained**:
- **Race-free** by construction. No mutex/flock needed.
- **Queryable**. `jq` / `Where-Object` against JSONL is trivial.
- **Replayable**. Reconstruct any past state from the event stream.
- **Sparse reads**. An agent can read only events `since=<ts>` or `to=me`.
- **Causal references**. Every event has a stable `id`; downstream events
  reference it (no more "second-line ambiguity").

**Open trade-offs / risks**:
- More moving parts than v1 (tools/render needed before reads → mitigated
  by lazy regeneration on read).
- JSONL is less inviting for humans to skim than markdown → mitigated by
  the rendered views in `view/`.
- The `state/issues/<id>.json` IS mutable. If two agents both claim it,
  there's still a window. → Mitigated by the protocol rule: only the
  current `assigned-to` writes; re-assignment is an event in the journal,
  not a direct file write.

## Iteration 3 — MCP server + SQLite (current)

**Design**:

1. **`agent-comms/state.sqlite`** — single SQLite database, WAL mode.
   Schema: `events` (append-only), `issues` (mutable per-resource),
   `inbox_reads` (per-agent read receipts), `issue_counters` (id sequence).
2. **`mcp-server/`** — a Node MCP server (single runtime dep:
   `@modelcontextprotocol/sdk`, plus `zod` for input validation;
   `node:sqlite` is built-in to Node ≥ 22.5, so zero native deps).
   Exposes 12 typed tools (see README / PROTOCOL).
3. **Agents never touch files for comms**. They call MCP tools. The
   server validates inputs, enforces the state machine and ownership
   rules, and writes inside SQL transactions.

**Borrowed from**:
- **MCP** (Anthropic): the standard for tool-based extension of Claude
  agents. 97M+ monthly SDK downloads in 2026, native plumbing for Claude
  Code.
- **A2A** (Google): task state machine (`submitted` / `working` /
  `input_required` / `blocked` / `completed` / `failed` / `canceled`),
  capability cards.
- **LangGraph state model**: typed, validated state mutations through a
  controlled interface (Zod schemas at the tool layer).
- **Event sourcing**: append-only `events` table preserved from v2.

**Properties gained over v2**:
- **ACID atomicity**: `open_issue` writes the state row AND the
  `kind=issue` event in *one* transaction. The v2 drift risk between
  the journal event and the JSON state file is gone.
- **Real validation**: every write goes through a Zod schema; invalid
  inputs are rejected before they hit the DB.
- **State machine enforced server-side**: invalid transitions
  (`submitted -> completed` skipping `working`) error out; ownership
  rule (`only assigned_to may mutate`) error out.
- **Concurrency safety**: SQLite WAL mode + `BEGIN IMMEDIATE`; the smoke
  test fires 50 parallel writes from 5 agents — zero conflicts.
- **Sparse onboarding**: `state_summary` returns ONLY what the agent needs
  for its turn (unread count, your open issues, last 10 events). No more
  re-reading a full markdown log every turn.
- **Queryable**: SQL is the right tool for "all critical issues assigned
  to me" — no markdown parsing.
- **Native Claude Code integration**: registered via `.mcp.json`; tools
  appear automatically.

**Smoke test (`mcp-server/smoke-test.js`)** verifies, in order:
1. `initialize` + `tools/list` (12 tools registered).
2. Empty `state_summary` → 0 unread.
3. `open_issue` atomic write (state row + event in one txn).
4. `record_file_change`, `send_message`, `log`.
5. Recipient `state_summary` shows the unread + the assigned issue.
6. `transition_issue` claim transitions `submitted → working` and takes ownership.
7. Unauthorized agent's `transition_issue` is rejected.
8. Invalid state-machine jump (`working -> submitted`) is rejected.
9. Proper `resolve` transitions to `completed`.
10. **50 parallel writes across 5 agents** — all succeed, no lost writes.
11. Final event count matches expectations.

All 14 assertions pass. v3 is production-ready for the local Claude Code
use case.

## Iteration 4 — Phase-gates, dependencies, Agent Teams coexistence (current)

Trigger: comparison with Anthropic's experimental
[Agent Teams](https://code.claude.com/docs/en/agent-teams) feature.
Conclusion: Agent Teams is a runtime/transport layer (independent
teammate processes, mailbox, lifecycle hooks); v3's MCP+SQLite is the
durable system of record. They compose. Borrow the three concrete
improvements the comparison surfaced.

**Changes shipped**:

1. **Phase-gate enforcement in `transition_issue`.** A `resolve` on an
   issue whose `area` is `qa` or `security` requires a matching `signoff`
   event with `verdict='approved'`. Enforced *inside the SQL transaction*
   in [mcp-server/db.js](mcp-server/db.js) — hooks can be skipped, ACID
   cannot. Resolves the "Iteration 4 future" candidate from earlier.

2. **`depends_on` column on `issues`.** Optional array of `ISS-ids`.
   `resolve` is rejected while any dependency is not in state `completed`.
   Existence of referenced ids is validated at `open_issue` and at
   dependency updates. Mirrors Agent Teams' built-in dependency tracking
   but inherits our state machine + ACID.

3. **Schema migration story.** Added `PRAGMA user_version` and an
   in-place migration path in `openDb()` — `v0` databases get
   `ALTER TABLE issues ADD COLUMN depends_on TEXT` applied at next open,
   preserving existing rows. Verified end-to-end. Needed *now*, before
   more state accretes.

4. **`getIssue.history`** merges `kind='issue'` and `kind='signoff'`
   events into one chronological timeline (was issue-only).

5. **Plan-approval convention** documented in
   [agent-comms/PROTOCOL.md](agent-comms/PROTOCOL.md). No new tool —
   reuses `open_issue({area:'arch'})` + `signoff` primitives.

6. **README "Running with experimental Agent Teams"** section explains
   the coexistence story (different storage paths, complementary layers).

**Explicitly deferred** (with reasoning):

- **`wait_for_message` long-poll tool** — would burn tokens in a holding
  pattern inside a synchronous subagent. Re-evaluate only when running
  under Agent Teams with persistent parallel teammates.
- **Dedicated plan-approval workflow tool** — the existing
  `signoff({verdict:'approved'})` primitive plus a documented convention
  covers it without ceremony.
- **Hook-based phase-gate enforcement** — hooks (`TeammateIdle`,
  `TaskCompleted`) are good for UX nudging but cannot be the authoritative
  gate; the SQL transaction is.

**Smoke test** ([mcp-server/smoke-test.js](mcp-server/smoke-test.js)) now
asserts 18 invariants including: phase-gate blocks a `qa` resolve without
signoff, the happy path with signoff succeeds, `getIssue.history` includes
signoffs, dependency blocks resolve until dep is completed, dependency
satisfied allows the dependent to resolve, 50 parallel writes succeed.

## Iteration 5+ — Future candidates

- **Subscription/notifications**: a `wait_for_message(agent)` tool that
  long-polls — only sensible under Agent Teams.
- **Multi-instance**: SQLite → Redis or Postgres if cross-process state
  emerges (e.g., multiple Claude Code instances on the same workspace).
- **Team memory**: a `decisions` table to pin durable architectural
  choices that survive across sessions.
- **Cost-aware routing**: track per-agent token usage; manager queries
  "cheapest agent that can do X".
- **OpenTelemetry export**: every tool call emits a span; pipe to a local
  collector for replay/analysis.
- **Settings hooks as UX companion** (not enforcement): wire
  `TeammateIdle` to surface blocked/critical issues to the lead's prompt
  via a small read-only script that calls `state_summary`.
